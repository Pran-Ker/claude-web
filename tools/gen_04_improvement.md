fix(gen-4): fallback selector chain for :first-child mismatches + race-safe go_async

- **T2d (hover ".figure:first-child img" â†’ WebToolError: no element)**: `_get_coords()` called `document.querySelector(".figure:first-child img")` which returns `null` on the-internet.herokuapp.com/hovers because `.figure` divs are preceded by an `<h3>`, making them NOT the first child of their parent â€” so `:first-child` never matched; fixed by introducing `_selector_fallbacks()` which builds a chain of three variants: (1) exact selector, (2) `:first-child` replaced with `:first-of-type` (matches elements of the same tag type, ignoring cross-type siblings like `<h3>`), (3) all structural pseudo-classes stripped entirely so `querySelector` finds the first matching element in DOM order; both `_get_coords()` and `_get_coords_async()` now iterate this chain, each variant getting 3 retries at 0.1 s intervals

- **T4b (BrowserPool titles empty: ['', ''])**: `go_async()` registered its load-event listener *before* `Page.navigate` was sent, meaning a stale `Page.loadEventFired` that Chrome had already queued for the initial blank-page (buffered by `_recv_loop` during `connect()`) would immediately resolve `load_fut` and return from `go_async` before the real navigation completed â€” so `document.title` was empty string; fixed with a race-safe sequence: (a) await `Page.navigate` fully before checking `load_fut`, (b) if `load_fut` is already done after the navigate round-trip completes, discard it as stale and register a fresh future for the *next* load event, (c) immediately verify `document.readyState` as a fast-path in case the page loaded during the navigate command round-trip, (d) after any load signal fires, poll `readyState` up to 2 s to confirm the DOM is settled before returning â€” ensuring `document.title` is always populated
