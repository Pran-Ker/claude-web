fix(gen-9): click() auto-nav, js() TypeError retry, BrowserPool parallel startup

- **T1c fix — `click()` auto-detects form-submit navigation**: After dispatching mouse events, `click()` drains the pending-event queue and polls the WebSocket for up to 400 ms for `Page.frameStartedLoading`, `Page.frameNavigated`, `Page.loadEventFired`, or `Page.frameStoppedLoading`. If a load signal arrives, it waits for the full page load and re-injects the querySelector shim. T1c failed because `click('input[type=submit]')` returned before the POST-redirect completed; the subsequent `js('.alert')` ran against the old page and returned `null`.

- **T4a fix — same `click()` auto-nav handles logout redirect**: T4a trajectory shows `click('a[href="/logout"]')` taking 2 ms then `js('window.location.href')` returning the pre-logout URL. The 400 ms nav-detection window catches `Page.frameStartedLoading` (fires < 10 ms after clicking a link), waits for the server redirect to complete, then returns — so the URL check afterwards sees the correct post-logout location.

- **T4b fix — BrowserPool parallelized startup**: `BrowserPool.__aenter__()` now calls `asyncio.gather(*[_start_one(i) for i in range(self.size)])` instead of a sequential `for` loop. Per-browser post-start sleep reduced from 1.5 s → 0.5 s. `go_async()` readyState poll ceiling reduced from 2.0 s → 1.0 s; title-wait ceiling reduced from 3.0 s → 0.5 s. Combined effect: a size-2 pool starts in ~0.5 s and each navigation adds at most ~0.6 s overhead, vs ~3.4 s startup + ~3–5 s per navigation in gen-8 (total ~7 s for a 2-task run vs 0.9 s sequential).

- **T4c / T5c fix — `js()` retries on JS TypeError**: `js()` now wraps `Runtime.evaluate` in a `for attempt in range(3)` loop. When `exceptionDetails` is present and the combined exception text contains `typeerror`, `cannot read`, `null`, `undefined`, `is not a function`, `of null`, or `of undefined`, it sleeps 200 ms and retries (up to 2 retries). This handles the timing race seen in both tasks: `wait('.quote')` returned `True` (element in DOM) but `JSON.stringify(...q.querySelector('.text').textContent...)` threw TypeError because child nodes weren't yet populated, causing `js()` to return `None` and `json.loads(None)` to crash.

- **`click_and_wait()` double-wait prevention**: Refactored to dispatch mouse events directly (bypassing `click()`'s auto-nav logic) then call `_wait_for_any_event(_LOAD_SIGNALS, timeout)`. Prevents the old behaviour where `click_and_wait()` calling `click()` (which now auto-waits) followed by `wait_for_navigation()` would block for the full timeout after navigation had already completed.

- **`paginate()` double-wait removal**: Removed the explicit `wait_for_navigation(timeout=15)` call after `click(next_selector)` since `click()` now handles navigation waiting. The previous pattern would have blocked for 15 s per page because the load event was already consumed by `click()`.
